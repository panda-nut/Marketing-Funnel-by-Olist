# Olist 巴西电商数据操作与练习说明

## 1. 先明确：这个项目真正要练的是什么

这个项目最值钱的地方，不是“数据大”，也不是“把 CSV 导进 MySQL”本身，而是你能不能围绕业务问题写出靠谱的 SQL。

这份数据集非常适合练关系型分析，因为它天然不是单表场景：

- 一个订单对应一个 `order_id`
- 一个订单可能有多个 `item`
- 每个 `item` 可能由不同 `seller` 履约
- 同一个订单还会关联支付、物流、评论、客户位置、商品属性等信息

这意味着你做分析时，必须时刻想清楚当前的分析粒度到底是：

- 订单粒度
- 订单商品粒度
- 用户粒度
- seller 粒度
- 商品品类粒度

如果粒度没想清楚，`JOIN` 一多，指标就很容易被重复计算。

## 2. 为什么它天然是关系型多表分析场景

这套数据不是那种“一张明细表直接算完”的练习数据，而是典型的电商业务数据模型。

你通常会围绕下面这些表来分析：

- `olist_orders_dataset`：订单状态、下单时间、发货时间、签收时间、预计送达时间
- `olist_order_items_dataset`：订单内每个商品项、对应 `seller`、商品价格、运费
- `olist_order_payments_dataset`：支付方式、分期数、支付金额
- `olist_order_reviews_dataset`：评分、评论时间、评论内容
- `olist_customers_dataset`：客户唯一标识、州、城市
- `olist_products_dataset`：商品品类、重量、长宽高、描述长度、图片数
- `product_category_name_translation`：葡语品类到英语品类的映射
- `olist_sellers_dataset`：卖家所在州、城市
- `olist_geolocation_dataset`：邮编前缀对应经纬度

一个很重要的认知是：

- `orders` 更接近订单主表
- `order_items` 是订单商品明细表
- `payments` 是订单支付明细表
- `reviews` 是订单评价表

所以做题时不要把所有表无脑直接连在一起。  
比如 `orders` 同时连接 `order_items` 和 `payments` 时，就可能出现一笔订单被乘出来多行的问题。

## 3. 练最有价值的 SQL

这个项目最值得练的，不是背语法，而是你能不能从业务问题出发，自己判断应该连哪些表、用什么口径、按什么粒度聚合。

这些主题都和 Kaggle 对该数据集的描述非常契合：支付、物流表现、产品质量、评论、客户位置。

### 经营概览

- 总订单数
- 总 GMV
- 平均客单价
- 月度订单趋势

这组题目主要训练你：

- 会不会区分 `COUNT(*)` 和 `COUNT(DISTINCT order_id)`
- 会不会定义 GMV 口径
- 会不会按月做时间聚合
- 会不会处理订单状态筛选

建议你在 GMV 口径上主动做两版：

- 商品金额口径：`SUM(price)`
- 含运费口径：`SUM(price + freight_value)` 或参考 `payment_value`

### 用户分析

- 复购用户占比
- 首单与最近一次下单
- 不同州/城市客户分布

这组题目主要训练你：

- 会不会区分 `customer_id` 和 `customer_unique_id`
- 会不会使用分组聚合和窗口函数
- 会不会做用户生命周期分析

其中 `customer_unique_id` 更适合做“真实用户”的复购分析，`customer_id` 更像一次订单收货身份。

### 商品与商家分析

- 热销品类 Top 10
- 不同 seller 的销量和销售额
- 平均运费最高的商品类别

这组题目主要训练你：

- 会不会把 `order_items`、`products`、`category_translation`、`sellers` 串起来
- 会不会处理商品粒度和订单粒度的差异
- 会不会做 Top N 排名

### 支付与履约分析

- 支付方式分布
- 延迟交付率
- 配送时长与评论分数的关系

这组题目主要训练你：

- 会不会做条件聚合
- 会不会计算时间差
- 会不会把履约数据和评论数据结合起来看

例如：

- 延迟交付率可以比较 `order_delivered_customer_date` 和 `order_estimated_delivery_date`
- 配送时长可以用下单到签收的日期差来衡量
- 再把配送时长和 `review_score` 关联，就能看物流体验是否影响评分

## 6. 这份数据最容易踩的坑

- 不要把订单数直接按 `order_items` 表来数，否则会把一笔多商品订单重复计算
- 不要把 `orders`、`order_items`、`payments` 直接全量连起来后再求和，否则金额可能被放大
- 做用户复购时优先看 `customer_unique_id`，不要只看 `customer_id`
- 做履约分析时要先考虑是否只保留 `delivered` 订单
- 做品类分析时最好连上 `product_category_name_translation`，否则品类名称不够直观
- 做时间趋势时要先统一统计口径，比如到底统计全部订单、已支付订单，还是已签收订单

### 订单金额对账检查

为验证数据导入后的金额一致性，分别从 `olist_order_items_dataset` 计算订单商品+运费金额，从 `olist_order_payments_dataset` 计算订单支付金额，并在订单粒度进行比对。

对账结果显示，共有 98,665 笔订单可参与比对，其中 98,362 笔订单的金额差异绝对值不超过 0.01，一致率达到 99.69%，说明整体数据质量较高、金额口径基本可靠。

其余 303 笔订单存在明显差异，占比约 0.31%。异常订单按差异金额分布如下：

- 0.01~1：54 笔
- 1~5：77 笔
- 5~10：74 笔
- 大于 10：98 笔

可以判断，少量异常不仅包含小额舍入误差，也包含部分较大金额偏差订单。因此在后续分析中，应将 `order_items` 金额口径与 `payments` 支付口径区分使用：前者更适合用于商品销售额分析，后者更适合用于支付行为分析，而差异订单则作为数据审计样本单独保留。

>对于金额的审查（如果存在多表间的金额关系，应当执行对账）

```sql
WITH item_amount AS (
    SELECT
        order_id,
        SUM(price + freight_value) AS item_total
    FROM olist_order_items_dataset
    GROUP BY order_id
),
payment_amount AS (
    SELECT
        order_id,
        SUM(payment_value) AS payment_total
    FROM olist_order_payments_dataset
    GROUP BY order_id
)
SELECT
    i.order_id,
    i.item_total,
    p.payment_total,
    ROUND(i.item_total - p.payment_total, 2) AS diff_amount
FROM item_amount i
JOIN payment_amount p
    ON i.order_id = p.order_id
WHERE ABS(i.item_total - p.payment_total) > 0.01
LIMIT 20；
```

### 业务拆解

>分析GMV的变化到底是源自客户数的增多还是客单价变化

```sql
SELECT
    month,
    ROUND(gmv, 2) AS gmv,
    paid_orders,
    buyers,
    ROUND(gmv / paid_orders, 2) AS avg_order_value,
    ROUND(paid_orders / buyers, 2) AS purchase_frequency,
    ROUND(gmv / buyers, 2) AS gmv_per_buyer,
    ROUND(
        (gmv - LAG(gmv) OVER (ORDER BY month))
        / LAG(gmv) OVER (ORDER BY month) * 100,
        2
    ) AS gmv_mom_pct
FROM monthly_panel
ORDER BY month;
```
>结构分析：先看品类，再看地区

- 品类变化：哪一个category拉动变化（order_items.price 来看商品销售结构）
- 地区变化：哪个州或者是城市的表现强
