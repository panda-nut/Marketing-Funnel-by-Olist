# 1. 整个数据导入流程

**常见问题：**

- 你怎么把 CSV 导入 MySQL？
- 为什么不用手工导入，而是用 Python？
- 整个流程分几步？

**标准回答：**

`CSV -> DataFrame -> MySQL 表`

1. 先用 Python 的 `pandas.read_csv()` 读取 CSV 文件，得到 `DataFrame`。
2. 再用 `SQLAlchemy + pymysql` 连接 MySQL。
3. 最后用 `df.to_sql()` 把 `DataFrame` 写入 MySQL 的目标表。

如果数据量比较大，还可以加上分块读取、字段清洗、类型转换和异常处理。

# 2. CSV、DataFrame、MySQL 三者的关系

**常见问题：**

- `read_csv()` 之后的数据存到哪里？
- `to_sql()` 做了什么？
- CSV 和数据库表有什么区别？

**标准回答：**

- CSV 是磁盘上的文本文件。
- DataFrame 是 Python 内存中的表格对象。
- MySQL 表是数据库中的持久化存储结构。

可以这样理解：

- `read_csv()` 是把磁盘上的 CSV 解析后读入内存，变成 `DataFrame`。
- `to_sql()` 是把内存中的 `DataFrame` 再写入数据库表。

从内存角度看：

- CSV 本身是文本文件，占用的是磁盘空间。
- 读入后会变成内存中的 `DataFrame`，通常比原始 CSV 更占内存，因为还包含索引、对象引用和类型开销。
- 执行 `to_sql()` 时，`DataFrame` 仍在内存中，同时 Python 和 MySQL 还可能有批量写入缓冲，因此导入时的峰值内存通常会高于 `DataFrame` 本身。
- 如果文件很大，可以使用 `chunksize` 分块读取或分批写入，控制内存峰值。

# 3. 为什么要先建立数据库

**常见问题：**

- 为什么要先 `CREATE DATABASE`？
- 不建数据库可以直接导入吗？

**标准回答：**

不能直接“随意导入”，因为 MySQL 中的表必须属于某个数据库。  
所以通常要先创建数据库，再连接到该数据库后导入表。

# 4. `to_sql()` 的常见参数

示例：

```python
df.to_sql("accounts", con=engine, if_exists="replace", index=False)
```

常见参数说明：

- `if_exists="replace"`：如果表已存在，就先删表再重建。
- `if_exists="fail"`：如果表已存在，就直接报错。
- `if_exists="append"`：向已有表追加数据。

`index=False` 的含义：

DataFrame 默认有索引列，但这个索引通常不是原始业务数据的一部分。  
导入 MySQL 时一般不希望把默认索引也写成一列，所以通常设为 `False`。

# 5. 为什么 Python 比纯 SQL 导入更适合

**常见问题：**

- 为什么不用 `LOAD DATA INFILE`？
- 为什么选择用 Python 导入？

**标准回答：**

如果只是导入单个、规则比较统一的小 CSV，`LOAD DATA INFILE` 或 `LOAD DATA LOCAL INFILE` 确实可能更高效。  
但如果面对多个文件，或者导入前还要做清洗、重命名、类型转换、缺失值处理、异常处理和自动化流程控制，那么 Python 会更灵活。

可以概括为：

- SQL 导入更偏向高性能、直接入库。
- Python 导入更偏向灵活处理和自动化。

关于 `LOCAL INFILE` 的常见问题：

- 可能受到服务端配置、客户端配置、用户权限等限制。
- 有时还会受到 `secure-file-priv` 等安全策略影响。
- 路径也更容易出问题，比如客户端和服务端对文件位置的理解不同，或者 Windows 路径、中文路径、绝对路径写法不一致。

# 6. 数据类型映射

**常见问题：**

- CSV 的字段类型怎么变成 MySQL 的字段类型？
- `to_sql()` 会自动建表吗？
- 自动推断类型会有问题吗？

**标准回答：**

`to_sql()` 可以自动建表，并根据 `DataFrame` 的列类型推断 MySQL 字段类型。  
但自动推断不一定总是最优，尤其是日期、金额、布尔值、长文本这类字段，实际项目里常常会：

- 先手动清洗和转换类型；
- 手动指定字段类型；
- 或者先手动建表，再把数据写进去。

# 6.1 遇到 schema 不匹配怎么解决

**常见问题：**

- 为什么 `to_sql(if_exists="replace")` 之后，字段类型不对了？
- 为什么主键、外键、`DEFAULT` 没有建出来？
- 如果自动建表不可靠，实际项目里怎么修？

**标准回答：**

这次我实际遇到的问题是：程序流程本身没有问题，`CSV -> DataFrame -> MySQL` 这条链路是通的，但导入层的 schema 丢了。  
根本原因不是 CSV 没读出来，而是我一开始直接用 `df.to_sql(..., if_exists="replace")` 让 pandas / SQLAlchemy 自动建表。  
这种方式能把数据写进去，但它更适合“快速落表”，不适合“严格还原数据库结构”，因为：

- 它主要根据 `DataFrame` 当前列类型推断字段类型；
- 它不会自动帮我还原业务主键、外键约束；
- 它也不会把我想要的 `DEFAULT NULL`、索引、复合主键这类 schema 信息完整保留下来。

所以这次修复不是改导入流程，而是把“自动建表”改成了“显式建 schema + 再导入数据”。

可以概括成：

`CSV -> DataFrame -> 类型清洗/转换 -> 手动 CREATE TABLE -> to_sql(append) 导入数据`

1. 先读取 CSV，但不再依赖 `to_sql()` 自动推断整张表结构。
2. 针对每张表先做字段清洗和类型转换，比如：
   - 时间列先用 `pd.to_datetime()` 转成 `DATETIME` / `DATE`
   - 金额列转成 `DECIMAL`
   - 计数字段转成 `INT` / `SMALLINT`
   - 布尔字段先映射成 `True / False`
   - 缺失值统一转成 `NULL`
3. 再手动执行 `CREATE TABLE`，把主键、外键、索引和 `DEFAULT NULL` 明确写出来。
4. 最后再用 `to_sql(if_exists="append")` 只负责“写数据”，而不是“决定表结构”。

这次实际改动里，重点解决了三类 schema 问题：

- 数据类型匹配：
  原来有些列容易被当成 `text` 或宽松字符串；修复后把日期、金额、整数、布尔值都先转好，再落到 MySQL 对应类型。
- KEY 键值匹配：
  原来自动建表不会建立业务主键和外键；修复后按真实业务关系补了主键、复合主键和可落地外键。
- DEFAULT / NULL 匹配：
  原来只是一张普通表，没有明确的默认值定义；修复后在 DDL 里把允许为空的列写成 `DEFAULT NULL`，让 MySQL 中的结构更完整。

这里还有一个很重要的工程判断：

不是所有“看起来像外键”的字段都能强行建外键。  
我在导入前专门检查了父子表关系，发现有些源数据本身就存在孤儿值，如果硬建外键反而会导致导入失败。  
所以实际项目里不能只看表关系图，还要先做数据校验，再决定哪些约束可以真正落库。

面试里如果被问到“你是怎么解决 schema 不匹配的”，比较合适的回答方式是：

我最开始用 `to_sql(replace)` 做快速导入，流程是通的，但自动建表导致 schema 丢失，主要表现为字段类型不准确、主键外键没有建立、`DEFAULT` 没有落库。  
后来我把方案改成“先显式建表，再 append 导入数据”：先用 pandas 做类型清洗和转换，再手动执行 `CREATE TABLE` 定义字段类型、主键、外键和默认值，最后用 `to_sql(append)` 只负责写数据。  
这样就把“数据导入”和“schema 管理”拆开了，导入结果会更可控，也更符合真实项目里的数据库建设方式。

# 7. 日期字段怎么处理

**常见问题：**

- CSV 里的日期字段怎么导入？
- 如果原始数据里是字符串格式日期怎么办？

**标准回答：**

可以在 `read_csv()` 之后先用 pandas 做日期转换，比如 `pd.to_datetime()`，再写入 MySQL。  
否则日期字段可能会被当成普通字符串存入数据库。

# 8. 缺失值的处理

**常见问题：**

- CSV 中有空值怎么办？
- MySQL 中会怎么存？

**标准回答：**

pandas 读取 CSV 后，缺失值通常会表现为 `NaN`。  
写入 MySQL 时，一般会转换为 `NULL`。  
正式导入前，最好先确认关键字段是否允许为空；如果不允许，就要提前清洗或填补。

# 9. 导入后如何验证数据是否正确

**常见问题：**

- 导入后怎么确认表建成功了？
- 怎么确认数据没有少？
- 怎么快速抽查内容？

**标准回答：**

- `SHOW TABLES;` 查看表是否创建成功。
- `SELECT COUNT(*) FROM 表名;` 对比 CSV 行数。
- `SELECT * FROM 表名 LIMIT 10;` 抽样检查数据内容。
